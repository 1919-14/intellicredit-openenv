text

├─ Agent approves borrower at T=3
├─ Borrower appears healthy for T=4 through T=9

├─ Hard constraints: NPA rate < 5%, CRAR > 12.5%
├─ Soft constraints: Sector exposure limits
├─ Breach = immediate episode termination
├─ Agent must balance yield vs safety
└─ Maps directly to RBI regulatory requirements

├─ All 25 features are causally dependent on archetype
├─ LLM agents can read features and they make logical sense
├─ Temporal linkage: macro shocks affect future companies
└─ This is what makes it LLM-evaluation-ready

INNOVATION 4: MULTI-OBJECTIVE OPTIMIZATION
├─ Objective 1: Maximize portfolio yield
├─ Objective 2: Minimize NPA rate
├─ Objective 3: Maximize capital utilization
├─ Objective 4: Maintain regulatory compliance

SECTION 4: THE CONSTRAINED MULTI-OBJECTIVE MDP DESIGN
4.1 MDP Formal Definition
text

FORMAL MDP SPECIFICATION:

State Space S:
├─ S ∈ ℝ⁴⁵ (45-dimensional continuous)
├─ Decomposed into 4 sub-spaces:
│  ├─ S_app ∈ ℝ²⁵ (current application features)
│  ├─ S_port ∈ ℝ¹⁰ (portfolio state)
│  ├─ S_macro ∈ ℝ⁵ (macro-economic indicators)
│  └─ S_alert ∈ ℝ⁵ (portfolio alert signals)
└─ All normalized to [0.0, 1.0] for RL stability

Action Space A:
├─ A = Discrete(3)
├─ a=0: APPROVE (full credit facility granted)
├─ a=1: CONDITIONAL_APPROVE (with covenants/monitoring)
└─ a=2: REJECT (application denied)

Transition Function T(s'|s,a):
├─ Deterministic component: Portfolio updates based on action
├─ Stochastic component: Macro-economic regime shifts
├─ Delayed component: NPA manifestation at T+k timesteps
└─ T is NOT simple — it has memory (portfolio carries forward)

Reward Function R(s,a,s'):
├─ Immediate component: Yield on approved loans
├─ Delayed component: NPA penalty at future timestep
├─ Constraint component: Regulatory breach penalty
├─ Shaping component: Exploration and correctness bonuses
└─ R ∈ [-5.0, +3.0] per step (wide range for learning)

Constraint Set C:
├─ c₁: NPA_rate < 0.05 (5% maximum)
├─ c₂: CRAR > 0.125 (12.5% minimum)
├─ c₃: Single_borrower_exposure < 0.15 (15% of capital)
├─ c₄: Sector_concentration < 0.30 (30% of portfolio)
└─ Violation of ANY constraint = episode termination

Discount Factor γ:
├─ γ = 0.99 (long horizon, delayed rewards matter)
└─ This is critical for NPA penalty propagation

Episode Length H:
├─ H = 12 timesteps (fixed)
├─ Each timestep: 1 application arrives
├─ Total decisions per episode: 10–15 (some timesteps have 0 or 2)
└─ Early termination possible on constraint violation
4.2 Why Constrained Multi-Objective MDP?
text

STANDARD MDP (What simple environments do):
├─ Single objective: maximize cumulative reward
├─ No constraints: agent can do anything
├─ No delayed consequences: reward is immediate
├─ No portfolio effects: each decision is independent
└─ Result: Toy problem, not research-grade

CONSTRAINED MULTI-OBJECTIVE MDP (What we do):
├─ Multiple objectives that CONFLICT:
│  ├─ Maximize yield (approve more → more revenue)
│  ├─ Minimize NPA rate (approve less → fewer defaults)
│  ├─ Maximize capital utilization (deploy capital efficiently)
│  └─ Maintain compliance (stay within regulatory limits)
│
├─ Hard constraints that TERMINATE episodes:
│  ├─ NPA rate exceeds 5% → episode over, massive penalty
│  ├─ Capital adequacy below 12.5% → episode over
│  └─ Agent must learn to self-regulate
│
├─ Delayed consequences that require FORESIGHT:
│  ├─ Approve at T=3 → default at T=10 → penalty propagates
│  ├─ Agent must learn temporal credit assignment
│  └─ This is fundamentally harder than immediate reward
│
└─ Result: Research-grade environment suitable for:
   ├─ Safe RL research (constraint satisfaction)
   ├─ Multi-objective optimization research
   ├─ Temporal credit assignment research
   ├─ Financial AI research
   └─ RBI/Basel regulatory compliance research

5.2 Timestep State Transition Logic
text

FOR EACH TIMESTEP T:

1. MACRO UPDATE
   ├─ Check if macro event scheduled for this T
   ├─ If yes: update macro state variables
   │  ├─ Adjust repo rate
   │  ├─ Update sector stress indices
   │  ├─ Modify GDP growth forecast
   │  └─ Shift credit cycle phase
   └─ Macro state affects all future application generation

2. APPLICATION GENERATION
   ├─ Generate new synthetic application
   ├─ Application features depend on:
   │  ├─ Hidden archetype (Tier A/B/C/D)
   │  ├─ Current macro state (affects DSCR, leverage)
   │  ├─ Sector assignment (affected by sector stress)
   │  └─ Random noise (stochastic variation)
   ├─ Compute hidden PD (Probability of Default)
   │  ├─ PD is NOT shown to agent
   │  ├─ PD determines if borrower will default later
   │  └─ PD influenced by archetype + macro + features
   └─ Generate forensic alerts (if applicable for this task)

3. OBSERVATION CONSTRUCTION
   ├─ Build 45-dimensional observation vector:
   │  ├─ Application features (25 dims): From generated application
   │  ├─ Portfolio state (10 dims): From running portfolio tracker
   │  ├─ Macro state (5 dims): From macro simulator
   │  └─ Portfolio alerts (5 dims): From NPA/stress monitoring
   └─ Normalize all values to [0.0, 1.0]

4. AGENT ACTION
   ├─ Agent receives observation
   ├─ Agent outputs action ∈ {0: APPROVE, 1: CONDITIONAL, 2: REJECT}
   └─ Action is validated (must be in valid set)

5. PORTFOLIO UPDATE
   ├─ If APPROVE:
   │  ├─ Add loan to portfolio
   │  ├─ Deduct capital (loan amount)
   │  ├─ Update sector exposure
   │  ├─ Schedule potential NPA at T+k (based on hidden PD)
   │  └─ Add yield to portfolio
   │
   ├─ If CONDITIONAL:
   │  ├─ Add loan with monitoring covenant
   │  ├─ Reduced yield (monitoring costs)
   │  ├─ Lower NPA probability (enhanced surveillance)
   │  └─ Update portfolio accordingly
   │
   └─ If REJECT:
      ├─ No portfolio change
      ├─ No yield earned
      ├─ No NPA risk added
      └─ Opportunity cost recorded

6. NPA CHECK
   ├─ Check if any previously approved loans default THIS timestep
   ├─ For each defaulting loan:
   │  ├─ Mark as NPA
   │  ├─ Compute loss amount
   │  ├─ Update NPA rate
   │  ├─ Deduct from capital
   │  └─ Apply delayed penalty to reward
   └─ NPA timing determined at approval time (hidden from agent)

7. CONSTRAINT CHECK
   ├─ Check NPA rate < 5%
   │  └─ If breached: TERMINATE episode, penalty = -5.0
   ├─ Check CRAR > 12.5%
   │  └─ If breached: TERMINATE episode, penalty = -5.0
   ├─ Check single borrower exposure < 15%
   │  └─ If breached: penalty = -1.0 (no termination)
   └─ Check sector concentration < 30%
      └─ If breached: penalty = -0.5 (no termination)

8. REWARD COMPUTATION
   ├─ Immediate reward components:
   │  ├─ Yield earned (if approved/conditional)
   │  ├─ Correctness bonus (if action matches optimal)
   │  ├─ Exploration bonus (forensic alignment)
   │  └─ Constraint penalties (if any breached)
   │
   ├─ Delayed reward components:
   │  ├─ NPA penalty (if past approval defaults now)
   │  └─ Stress signal penalty (if portfolio degrading)
   │
   └─ Total step reward = Σ(all components)

9. DONE CHECK
   ├─ If T == 12: done = True (natural end)
   ├─ If constraint violated: done = True (early termination)
   └─ If capital exhausted: done = True (can't approve more)

10. RETURN: (observation, reward, done, info)

6.2 Observation Normalization Strategy
text

NORMALIZATION RULES:

For continuous features (ratios, scores):
├─ Method: Min-Max normalization to [0, 1]
├─ Formula: x_norm = (x - x_min) / (x_max - x_min)
├─ Bounds defined per feature based on domain knowledge:
│  ├─ DSCR: min=0.0, max=3.0 → normalized
│  ├─ Current Ratio: min=0.0, max=4.0 → normalized
│  ├─ D/E Ratio: min=0.0, max=5.0 → normalized
│  ├─ CIBIL Score: min=300, max=900 → normalized
│  └─ Revenue: min=0, max=100Cr → normalized
└─ Clipping: Values outside bounds clipped to [0, 1]

For binary features (flags):
├─ Method: Direct {0, 1} encoding
├─ 0 = Not present / No alert
└─ 1 = Present / Alert active

For count features (litigation, charges):
├─ Method: Sigmoid-based normalization
├─ Formula: x_norm = sigmoid(x - threshold)
├─ This creates smooth transition around threshold
└─ Example: 3 litigations → sigmoid(3-2) = 0.73

├─ When agent APPROVES at timestep T:
│  ├─ Compute hidden PD for borrower
│  ├─ Schedule potential NPA at T + delay_period
│  ├─ delay_period = random(3, 8) timesteps
│  └─ NPA probability = hidden_PD × macro_stress_multiplier
│
├─ At timestep T + delay_period:
│  ├─ Roll dice: if random() < NPA_probability:
│  │  ├─ Borrower defaults!
│  │  ├─ R_delayed_npa = -2.0 (massive penalty)
│  │  ├─ Loss amount = loan_amount × (1 - recovery_rate)
│  │  ├─ Portfolio NPA rate increases
│  │  └─ Capital adequacy decreases
│  └─ If random() ≥ NPA_probability:
│     └─ Borrower performing (no penalty)
│
├─ CONDITIONAL approvals:
│  ├─ NPA probability reduced by 30%
│  ├─ If NPA occurs: penalty = -1.5 (less than full approve)
│  └─ Monitoring covenant partially protects
│
└─ KEY INSIGHT:
   ├─ This creates temporal credit assignment problem
   ├─ Agent must learn to PREDICT future defaults
   ├─ Good decisions at T=3 prevent pain at T=10
   └─ This is what makes the environment research-grade

├─ Threshold: CRAR > 12.5%
├─ Formula: CRAR = (Tier1_Capital + Tier2_Capital) / Risk_Weighted_Assets
├─ Simplified: CRAR = remaining_capital / (deployed_capital × risk_weight)
├─ Violation: EPISODE TERMINATION + R = -5.0

├─ Why It Matters:
│  ├─ Banks with low CRAR can't lend further
│  ├─ Agent must preserve capital buffer
│  └─ Over-lending leads to constraint breach
└─ Detection: Checked AFTER each approval action

CONSTRAINT 3: SINGLE BORROWER EXPOSURE
├─ Threshold: No single borrower > 15% of total capital
├─ Formula: exposure = loan_amount / total_capital
├─ Violation: PENALTY -1.0 (no termination)
├─ RBI Reference: Large exposure framework
├─ Why It Matters:
│  ├─ Concentration risk in single borrower
│  ├─ Agent must diversify across applications
│  └─ Forces portfolio thinking, not individual thinking
└─ Detection: Checked BEFORE each approval action

CONSTRAINT 4: SECTOR CONCENTRATION

├─ Formula: sector_pct = sector_deployed / total_deployed
├─ Violation: PENALTY -0.5 (no termination)

├─ Why It Matters:
│  ├─ Sector collapse affects all loans in that sector
│  ├─ Agent must diversify across sectors

10.1 Why Random Data Fails for LLM Evaluation
text

THE PROBLEM WITH RANDOM GENERATION:

Random approach:
├─ Generate 25 random floats
├─ Example: revenue=₹50Cr, DSCR=0.3, CIBIL=850, net_worth=-₹10Cr
├─ Problem: This is INCOHERENT
│  ├─ How can a company have ₹50Cr revenue but DSCR of 0.3?
│  ├─ How can CIBIL be 850 with negative net worth?
│  ├─ These numbers tell no logical story
│  └─ An LLM reading this will HALLUCINATE or make random decisions
└─ Result: LLM-based evaluator scores poorly, environment seems broken

Our causal approach:
├─ Generate hidden archetype FIRST (Tier A/B/C/D)
├─ ALL features derived from archetype + noise
├─ Example Tier A: revenue=₹50Cr, DSCR=1.8, CIBIL=780, net_worth=₹15Cr
│  ├─ Revenue is high → DSCR is good (cash flows support debt)
│  ├─ CIBIL is high → net worth is positive (consistent history)
│  ├─ All numbers tell SAME story: "Strong company"
│  └─ LLM reads this and understands: "This should be approved"
└─ Result: LLM-based evaluator works correctly, environment is credible

12.1 The Source of Truth Architecture

┌──────────────────────────────────────────────────────────────┐
│           GROUND TRUTH & HIDDEN STATE ENGINE                 │
└──────────────────────────────────────────────────────────────┘

KEY CONCEPT: "WE ARE GOD IN THIS UNIVERSE"

The environment knows everything. The agent sees only the observation.

WHAT THE AGENT SEES (45 dimensions):
├─ Application features (normalized)
├─ Portfolio state
├─ Macro indicators
└─ Portfolio alerts

WHAT THE AGENT DOES NOT SEE (Hidden State):
├─ Hidden credit tier (A/B/C/D)
├─ True Probability of Default (PD)
├─ NPA schedule (when default will occur)
├─ Recovery rate (how much capital is lost)
├─ Hard rule triggers (before agent's decision)
├─ Optimal action for this application
└─ Future macro events

HOW THE ENVIRONMENT USES HIDDEN STATE:

1. REWARD COMPUTATION:
   ├─ Agent says "APPROVE"
   ├─ Environment checks: hidden_PD = 0.65 (Tier D)
   ├─ Optimal action was "REJECT"
   ├─ Reward = -1.5 (dangerous approval)
   └─ Also schedules NPA at T+5

2. NPA DETERMINATION:
   ├─ At T=3, agent approved borrower with hidden_PD = 0.40
   ├─ Environment schedules NPA check at T=8
   ├─ At T=8: roll random(0,1)
   │  ├─ If random < 0.40: BORROWER DEFAULTS
   │  │  ├─ NPA event occurs
   │  │  ├─ Delayed penalty applied
   │  │  └─ Portfolio NPA rate increases
   │  └─ If random ≥ 0.40: BORROWER PERFORMS
   │     └─ No penalty (borrower survived)
   └─ This stochasticity is realistic AND learnable

3. HARD RULE DETECTION:
   ├─ Before presenting application to agent:
   │  ├─ Check if CIBIL < 550 → flag HR-01
   │  ├─ Check if net_worth < 0 → flag HR-04
   │  ├─ Check if wilful_defaulter → flag HR-03
   │  └─ [check all 6 hard rules]
   ├─ Hard rule flags stored in hidden state
   ├─ Some indicators ARE visible in observation
   │  (e.g., CIBIL score is visible, agent should infer)
   ├─ But the RULE is not explicitly stated
   └─ Agent must learn: "CIBIL < 550 = always reject"

4. OPTIMAL ACTION COMPUTATION:
   ├─ For each application, compute "optimal" action:
   │  ├─ If hidden_PD < 0.15: optimal = APPROVE
   │  ├─ If 0.15 ≤ hidden_PD < 0.40: optimal = CONDITIONAL
   │  ├─ If hidden_PD ≥ 0.40: optimal = REJECT
   │  └─ If hard_rule_triggered: optimal = REJECT
   ├─ This optimal is used for reward computation
   └─ Agent's score depends on matching optimal

SECTION 14: GRADER FUNCTIONS & SCORING FORMULAE
text

GRADER PROPERTIES (ALL TASKS):
├─ Output: float in [0.0, 1.0]
├─ Deterministic: Same episode → same score (always)
├─ Reproducible: Same agent + same seed → same score
├─ Transparent: Formula is documented and auditable
└─ Progressive: Task 1 score > Task 2 > Task 3 > Task 4 > Task 5

GRADER COMPONENT DEFINITIONS:

accuracy:
├─ Formula: correct_decisions / total_decisions
├─ correct = agent_action matches optimal_action
├─ Range: [0.0, 1.0]
└─ Higher is better

forensic_handling:
├─ Formula: correct_forensic_decisions / total_forensic_applications
├─ "correct" = reject/conditional when RED alerts present
├─ Range: [0.0, 1.0]
└─ Measures ability to recognize risk signals

hard_rule_compliance:
├─ Formula: correct_hard_rule_decisions / total_hard_rule_applications
├─ "correct" = REJECT when hard rule is triggered
├─ Range: [0.0, 1.0]
└─ Critical: Any non-reject on hard rule = compliance failure

constraint_satisfaction:
├─ Formula: 1.0 if no hard constraints breached, 0.0 if any breached
├─ Binary component (all-or-nothing for hard constraints)
├─ Range: {0.0, 1.0}
└─ Episode termination = 0.0 on this component

npa_avoidance:
├─ Formula: max(0, 1.0 - (npa_rate / target_npa_rate))
├─ target_npa_rate = 0.05 (5%)
├─ Range: [0.0, 1.0]
└─ NPA rate of 0% = 1.0, NPA rate ≥ 5% = 0.0

capital_efficiency:
├─ Formula: min(1.0, capital_deployed / (0.7 × total_capital))
├─ Optimal at 70% deployment
├─ Range: [0.0, 1.0]
└─ Too little deployment = low score

portfolio_health:
├─ Formula: (capital_remaining_pct) × (1 - npa_rate) × (yield_score)
├─ Range: [0.0, 1.0]
└─ Composite metric of overall portfolio quality

survival_rate:
├─ Formula: timesteps_survived / 12
├─ Range: [0.0, 1.0]
└─ Only relevant for Task 5 (where termination is likely)

missing_data_handling:
├─ Formula: 1.0 - (bad_approvals_with_missing / total_missing_apps)
├─ "bad" = approved an application with >3 missing critical features
├─ Range: [0.0, 1.0]
└─ Measures caution under uncertainty

text

┌──────────────────────────────────────────────────────────────┐
│              INFERENCE.PY ARCHITECTURE                       │
└──────────────────────────────────────────────────────────────┘

MANDATORY REQUIREMENTS:
├─ File name: inference.py (root directory)
├─ Uses: OpenAI Client (from openai import OpenAI)
├─ Reads: API_BASE_URL, MODEL_NAME, HF_TOKEN from env vars
├─ Runtime: < 20 minutes for all 5 tasks
└─ Output: Reproducible scores for each task

EXECUTION FLOW:

Step 1: Initialize Environment
├─ Import CreditAppraisalEnv from my_env.env
├─ Import OpenAI client
├─ Read environment variables
└─ Set random seed for reproducibility

Step 2: For Each Task (1 through 5):
├─ Set task: env.set_task(task_id)
├─ Reset environment: obs = env.reset()
├─ Initialize episode tracking variables
│
├─ Loop while not done:
│  ├─ Convert observation to text: prompt = build_prompt(obs)
│  ├─ Send prompt to LLM:
│  │  └─ response = client.chat.completions.create(
│  │       model=MODEL_NAME,
│  │       messages=[
│  │         {"role": "system", "content": SYSTEM_PROMPT},
│  │         {"role": "user", "content": prompt}
│  │       ],
│  │       temperature=0.1,
│  │       max_tokens=50
│  │     )
│  ├─ Parse LLM response to action:
│  │  └─ action = parse_action(response.choices[0].message.content)
│  │     ├─ "APPROVE" → 0
│  │     ├─ "CONDITIONAL" → 1
│  │     ├─ "REJECT" → 2
│  │     └─ Fallback: 2 (REJECT as safe default)
│  ├─ Step environment:
│  │  └─ obs, reward, done, info = env.step(action)
│  ├─ Track: actions, rewards, decisions
│  └─ Continue until done=True
│
├─ Compute task score via grader:
│  └─ score = env.grade()
├─ Print score:
│  └─ print(f"Task {task_id}: {score:.4f}")
└─ Continue to next task

Step 3: Output Summary
├─ Print all 5 scores
├─ Print average score
└─ Exit cleanly

SYSTEM PROMPT FOR LLM:
├─ "You are a senior credit officer evaluating MSME loan applications."
├─ "Analyze the Credit Appraisal Summary Sheet provided."
├─ "Make a decision: APPROVE, CONDITIONAL, or REJECT."
├─ "Consider: financial ratios, forensic alerts, portfolio state, macro environment."
├─ "If CIBIL < 550 or net worth is negative, ALWAYS REJECT."
├─ "If NPA rate is above 3%, be more conservative."
├─ "Respond with ONLY one word: APPROVE, CONDITIONAL, or REJECT."
└─ "Do not explain your reasoning."

ACTION PARSING (RegEx):
├─ Pattern: r"(APPROVE|CONDITIONAL|REJECT)"
├─ Case-insensitive matching
├─ Extract first match from LLM response
├─ If no match found: default to REJECT (safe)
└─ Map: APPROVE→0, CONDITIONAL→1, REJECT→2

SECTION 19: FASTAPI SERVER ARCHITECTURE
text

┌──────────────────────────────────────────────────────────────┐
│              FASTAPI SERVER ARCHITECTURE                     │
└──────────────────────────────────────────────────────────────┘

PURPOSE:
├─ Wrap gym.Env class in HTTP API
├─ Enable remote access from inference.py
├─ Enable hackathon validator to ping Space
└─ Serve on Hugging Face Spaces

HOW IT WORKS:

1. server/app.py calls create_app(CreditAppraisalEnv)
   └─ OpenEnv library auto-generates FastAPI endpoints

2. Endpoints created:

   POST /reset
   ├─ Triggers: env.reset()
   ├─ Returns: JSON { observation: [...], info: {...} }
   ├─ HTTP Status: 200 OK
   └─ Called by: inference.py at start of each task

   POST /step
   ├─ Accepts: JSON { action: 0 | 1 | 2 }
   ├─ Triggers: env.step(action)
   ├─ Returns: JSON {
   │    observation: [...],
   │    reward: float,
   │    done: bool,
   │    info: {...}
   │  }
   ├─ HTTP Status: 200 OK
   └─ Called by: inference.py at each timestep

   GET /state
   ├─ Triggers: env.state()
   ├─ Returns: JSON { state: {...} }
   ├─ HTTP Status: 200 OK
   └─ Called by: validator to check environment health

3. Server starts on port 7860 (HF Spaces default)

4. When hackathon validator pings Space URL:
   ├─ Sends GET to https://your-space.hf.space/state
   ├─ Expects HTTP 200 OK
   ├─ If received: environment is LIVE ✅
   └─ If not: DISQUALIFIED ❌

CONNECTION FLOW:

inference.py (Client)              HF Space (Server)
──────────────────                ─────────────────
                                  
env.reset()        ──HTTP POST──► /reset endpoint
                   ◄──JSON 200──  env.reset() called
                                  returns observation
                                  
env.step(action)   ──HTTP POST──► /step endpoint
                   ◄──JSON 200──  env.step(action) called
                                  returns (obs, r, d, info)
                                  
                                  (repeat 12 times per episode)
                                  (repeat 5 times for 5 tasks)
